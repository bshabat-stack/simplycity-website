Files in this folder whose names start with an underscore are ignored by the
blog, so this template never shows up on the site. It also lives outside the
locale folders, which is the other reason it can never be published by mistake.

## Each language has its own separate blog

Posts do not cross locales. There are three independent sets:

    content/blog/en/   ->  /blog/<slug>        (English)
    content/blog/es/   ->  /es/blog/<slug>     (Spanish)
    content/blog/he/   ->  /he/blog/<slug>     (Hebrew)

An English post is never listed on `/es/blog` or `/he/blog`, is never
auto-translated, and its URL under another locale returns a real 404. Write for
one audience, put the file in that audience's folder. If you want the same idea
in two languages, that is two posts, written separately, in two folders — they
stay unlinked, and each one can have whatever slug suits its own language.

The same slug in two folders is fine: `en/foo.md` and `he/foo.md` are simply two
unrelated posts.

## To publish a post

Copy this file into the locale folder for the audience you are writing for, and
name it after the URL you want — `why-reviews-matter.md` in `content/blog/en/`
becomes `/blog/why-reviews-matter`. Lowercase letters, numbers and hyphens only.
Fill in the frontmatter above and write the article below it in Markdown.

That is the whole process — no code changes, and nothing else to remember.
`title` is required: a post without one is treated as unpublished, so it stays
off the index, its URL 404s, and it is left out of the sitemap.

In particular, do NOT hand-add the post to a sitemap. `/sitemap.xml` is
generated by `api/sitemap.js`, which reads these folders on every request and
lists each post under its own locale only. Blog posts carry a self-referencing
canonical and deliberately no hreflang — hreflang would claim the post is a
translation of a post in another locale, which is exactly what it is not.

## Optional SEO frontmatter (all default gracefully)

`title`, `date` and `excerpt` are all a post needs. For a post that wants full
technical SEO, `api/blog-post.js` also reads these optional frontmatter keys and
renders them server-side into `<head>`; omit any and that tag is simply skipped:

    metaTitle:        # <title> + og:title + twitter:title (else: title)
    metaDescription:  # meta description + og/twitter description (else: excerpt)
    dateLabel:        # shown verbatim on the card + post header (else: formatted date)
    publishDate:      # article:published_time (else: date)
    updatedDate:      # article:modified_time
    author:           # article:author + byline name at the foot of the post
    authorBio:        # byline bio under the author name
    ogImage:          # site-root-relative path, e.g. /blog/<slug>/hero.jpg — made
                      # absolute for og:image / twitter:image, and preloaded
    ogImageAlt:       # og:image:alt

Frontmatter is still line-based `key: value` (no nested YAML) — one value per key.

## Optional structured-data sidecar `<slug>.schema.json`

Drop a `<slug>.schema.json` next to the post holding a JSON **array** of JSON-LD
objects (e.g. `Article`, `FAQPage`, `BreadcrumbList`). When present it is injected
verbatim and REPLACES the default `BlogPosting`; when absent the default is used.
A malformed sidecar is ignored (it falls back to `BlogPosting`, never errors).
`.json` files are not `.md`, so they never appear on the index or in the sitemap.
If a `FAQPage` block mirrors FAQs in the body, keep the two in sync when editing.
